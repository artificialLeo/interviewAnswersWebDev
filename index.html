DOM

The DOM is a cross-platform and language-independent interface that treats an HTML document as a tree structure wherein each node is an object representing a part of the document. The DOM represents a document with a logical tree.

DOM

DOM – это объектная модель документа, которую браузер создаёт в памяти компьютера на основании HTML-кода, полученного им от сервера. DOM – это набор связанных объектов, созданных браузером при парсинге её текста.

Doctype

DOCument TYPE — A code in an HTML page that informs the Web browser which version of the markup language was used to write the page in order to render it properly on screen. Typically the first line of code in the page, the DOCTYPE points to a Document Type Definition (DTD), which contains the markup rules.

Doctype

DOCTYPE — это определение типа документа. Благодаря этой записи, браузер определяет, какая в данном документе используется версия (Document Type Definition). Нужен он для того чтобы браузеры правильно отображали разметку документа. Если не указать DOCTYPE, то браузер будет добавлять "что то от себя", а это убивает кроссбраузерность.

default (package-private): (used in classes, constructors, interfaces, inner classes, methods and class fields) - Package level access. If the class is declared like this, it will only be available inside the package.

JS Prototype inheritance

In JavaScript, each object can have an association with another object - a so-called "prototype". If the search for a certain property (or method) in the original object fails, the interpreter tries to find the property of the same name (method) in its prototype, then in the prototype of its prototype, etc. For example, if we requested a call to object .property , JavaScript starts looking for this in the object itself, then in the object prototype, then in a prototype of the prototype, and so on until the end.

JS Прототипное наследование

В JavaScript каждый объект может иметь ассоциацию с другим объектом - так называемый "прототип". В случае, если поиск некоторого свойства (или метода) в исходном объекте заканчивается неудачно, интерпретатор пытается найти одноименное свойство (метод) в его прототипе, затем - в прототипе прототипа и т. д. К примеру, если мы затребовали обращение к object .property, JavaScript начинает искать свойство property в самом объекте object, затем – в прототипе object, прототипе прототипа object, и так до конца.

JS Closure

In simple words, this is an internal function, i.e. function in function.
A closure is called a closure because after the parent or outer function has been executed, the inner code still lives in the interpreter and can be executed. Errors will not occur even if the inner function (closure) uses variables from the outer function.

JS Promise

A Promise is an object that represents the final completion (or failure) of an asynchronous operation and its resulting value.
A promise has four states:
Pending is the initial state of the promise. The result of the promise is unknown because the operation has not completed.
Fulfilled - The asynchronous operation is complete and has a result.
Rejected - Asynchronous operation failed, cause exists.
Completed - Completed or Rejected.

The promise constructor takes resolve and reject as parameters. Resolve records the result of the operation, and reject records the reason for the failure of the operation. The result can be handled in the .then method, an error in the .catch method. The .then method also returns a promise, so we can use a chain of multiple .then.

JS Promise

Promise — это объект который представляет окончательное завершение (или сбой) асинхронной операции и ее результирующее значение.
У промиса есть четыре состояния:
Ожидание — начальное состояние промиса. Результат промиса неизвестен, поскольку операция не завершена.
Выполнено — асинхронная операция выполнена, имеется результат.
Отклонено — асинхронная операция не выполнена, имеется причина.
Завершено — выполнено или отклонено.

В качестве параметров конструктор промиса принимает resolve и reject. В resolve записывается результат выполнения операции, в reject — причина невыполнения операции. Результат может быть обработан в методе .then, ошибка — в методе .catch. Метод .then также возвращает промис, поэтому мы можем использовать цепочку, состоящую из нескольких .then.

How are HTTP and HTTPS different?

HyperText Transfer Protocol is an application layer data transfer protocol. HTTP is based on the "client-server" technology, that is, it is assumed that there are consumers (clients) that initiate a connection and send a request, and providers (servers) that wait for a connection to receive a request, perform the necessary actions and return a message with the result.
HyperText Transfer Protocol Secure is an extension to the HTTP protocol that supports encryption. Data transmitted over HTTPS is "packaged" in a cryptographic SSL or TLS protocol.

Чем отличаются HTTP и HTTPS?

HyperText Transfer Protocol - протокол прикладного уровня передачи данных. Основой HTTP является технология "клиент-сервер", то есть предполагается существование потребителей (клиентов), которые инициируют соединение и посылают запрос, и поставщиков (серверов), которые ожидают соединения для получения запроса, производят необходимые действия и возвращают обратно сообщение с результатом.
HyperText Transfer Protocol Secure - расширение протокола HTTP, поддерживающее шифрование. Данные, передаваемые по протоколу HTTPS, "упаковываются" в криптографический протокол SSL или TLS.

Design pattern

A design pattern is a common solution to a specific problem when designing a program architecture (a general concept for solving a particular problem, which will still need to be adjusted to the needs of your program).
It is like an engineering blueprint that draws a solution, but not specific steps to implement it. The implementation of one pattern may differ in two different programs.
Pattern descriptions consist of:
1) the problem that the pattern solves;
2) motivation to solve the problem in a way that the pattern suggests;
3) the structure of the classes that make up the solution;
4) an example in one of the programming languages;
5) implementation specifics in different contexts;
6) connections with other patterns.

Паттерн проектирования

Паттерн проектирования — это часто встречающееся решение определённой проблемы при проектировании архитектуры программ (общая концепция решения той или иной проблемы, которую нужно будет ещё подстроить под нужды вашей программы).
Он как инженерный чертёж, на котором нарисовано решение, но не конкретные шаги его реализации. Реализация одного паттерна может отличаться в двух разных программах.
Описания паттернов состоят из:
1) проблемы, которую решает паттерн;
2) мотивации к решению проблемы способом, который предлагает паттерн;
3) структуры классов, составляющих решение;
4) примера на одном из языков программирования;
5) особенностей реализации в различных контекстах;
6) связей с другими паттернами.

Polymorphism

Polymorphism is a property that allows the same name to be used to solve two or more similar but technically different problems. The goal of polymorphism, as applied to object-oriented programming, is to use a single name to define common actions for a class. The execution of each specific action will be determined by the data type. More generally, the concept of polymorphism is the idea of ​​"one interface, many methods". This means that you can create a common interface for a group of similar actions.

Полиморфизм

Полиморфизм — это свойство, которое позволяет одно и то же имя использовать для решения двух или более схожих, но технически разных задач. Целью полиморфизма, применительно к объектно-ориентированному программированию, является использование одного имени для задания общих для класса действий. Выполнение каждого конкретного действия будет определяться типом данных. В более общем смысле, концепцией полиморфизма является идея "один интерфейс, множество методов". Это означает, что можно создать общий интерфейс для группы близких по смыслу действий.

Inheritance

Inheritance is a process by which one object can acquire the properties of another object (inheritance of all properties of one object to another) and add features that are characteristic only of itself.

Наследование

Наследование это процесс благодаря которому один объект может приобрести свойства другого объекта (наследование всех свойств одного объекта другим) и добавлять черты характерны только для него самого.

Encapsulation

Encapsulation is hiding the implementation of a class and separating its internal representation from the external (interface), this is a property that allows you to close access to the fields and methods of a class to other classes, and give them access only through an interface (method). When using the object-oriented approach, it is not customary to use direct access to the properties of any class from the methods of other classes. To access the properties of a class, it is customary to use special methods of this class to get and change its properties.

Инкапсуляция

Инкапсуляция - это сокрытие реализации класса и отделение его внутреннего представления от внешнего (интерфейса), это свойство которое позволяет закрыть доступ к полям и методам класса другим классам, а предоставлять им доступ только через интерфейс (метод). При использовании объектно-ориентированного подхода не принято применять прямой доступ к свойствам какого-либо класса из методов других классов. Для доступа к свойствам класса принято задействовать специальные методы этого класса для получения и изменения его свойств.

Abstraction

Abstraction is the selection of the general characteristics of an object, excluding a set of insignificant ones. Using the principle of data abstraction, data is converted into objects. Data is processed as a chain of messages between individual objects. All objects exhibit their own unique behaviors. A huge plus of abstraction is that it decouples the implementation of objects from their details, which in turn allows high-level functions to be controlled through low-level functions.

Абстракция

Абстракция - это выделение общих характеристик объекта, исключая набор незначительных. С помощью принципа абстракции данных, данные преобразуются в объекты. Данные обрабатываются в виде цепочки сообщений между отдельными объектами. Все объекты проявляют свои уникальные признаки поведения. Огромный плюс абстракции в том, что она отделяет реализацию объектов от их деталей, что в свою очередь позволяет управлять функциями высокого уровня через функции низкого уровня.

Object

An object is a named model of a real entity that has specific property values and manifests its behavior, a named set of data (fields and properties of an object) that are physically located in the computer's memory and methods that have access to them. An object is a concrete instance of a class.

Объект

Объект — именнованная модель реальной сущности, обладающая конкретными значениями свойств и проявляющая свое поведение, обладающий именем набор данных (полей и свойств объекта), физически находящихся в памяти компьютера, и методов, имеющих доступ к ним. Объект — конкретный экземпляр класса.

GraphQL Aliases

Aliases help in a query to select the same field several times, but each time with a different argument. Aliases can be added to any field.

GraphQL Аллиасы

Аллиасы помогают в запросе выбрать одно и то же поле несколько раз, но каждый раз с другим аргументом. Псевдонимы могут быть добавлены в любое поле.

GraphQL fragment

GraphQL allows you to wrap subfields in a "fragment" and use them with the spread operator in selections. It is useful when we select the same fields over and over. The fragment is sent to the server along with the main activity in the HTTP request field.

GraphQL фрагмент

GraphQL позволяет обернуть «сабфилды» во "фрагмент" и использовать их со spread оператором в выборках. Полезно, когда мы выбираем одни и те же поля снова и снова. Фрагмент отправляется на сервер вместе с основной операцией в поле HTTP-запроса.

GraphQL variables

Using GraphQL variables reduces the amount of query code written. We declare them at the beginning of the request and are allowed to assign default values. Variables can also be nested.

GraphQL переменные

Использование переменных GraphQL уменьшает количество написанного кода запроса. Мы объявляем их в начале запроса, и им разрешено присваивать значения по умолчанию. Также переменные могут быть вложенными.

GraphQL interfaces

Interfaces in GraphQL schema allow us to create and use complex data types in an OOP manner. Types can implement multiple interfaces. When the return type is an interface, we can use the on keyword to conditionally access implementation-specific fields.

GraphQL интерфейсы

Интерфейсы в GraphQL схеме позвляют нам создавать и использовать комплексные типы данных на манер ООП. Типы могут реализовывать несколько интерфейсов. Когда возвращаемый тип является интерфейсом, мы можем использовать ключевое слово on для условного доступа к полям конкретной реализации.

GraphQL mutable operations

Mutable operations in GraphQL are used to modify data on the server side, such as creating an account, creating an authorization token, or adding records. Mutation can return values, we can pass arguments to mutation.

GraphQL мутабельные операции

Мутабельные операции в GraphQL используются для изменения данных на стороне сервера, например: создание учетной записи, создание токена авторизации или добавление записей. Мутация может возвращать значения, мы можем передать аргументы для мутации.

GraphQL Subscription

Instead of communicating through HTTP requests back and forth, GraphQL also has a Subscription operation type for making WebSocket connections, so the server can send updates to the client without the client's requests to the server.

GraphQL Subscription

Вместо общения путем HTTP запросов туда и обратно, GraphQL также имеет тип операции Subscription для создания WebSocket соединений, поэтому сервер может отправлять клиенту обновления без запросов клиента на сервер.
